# Feedback スキル

ユーザーからのフィードバック（要件の追加・変更・方針転換・優先度変更など）を受け取り、既存の GitHub Issue に反映します。

## 使い方

```
/feedback ログイン機能の件、ソーシャルログインはやっぱり使わない方向で。メール認証のみにしたい
/feedback Phase 1 に通知連動機能も含めたい
/feedback レビューで「エラーメッセージ統一ルール」を追加してほしいと言われた
/feedback docs/proposals/20260404_sample_proposal.md この提案書の内容をIssueに反映して
```

**引数は必須。** `$ARGUMENTS` が空の場合は使い方を案内して中断。

## 設計思想

フィードバックは開発中に頻繁に発生する。しかし口頭やチャットで受けたフィードバックは、Issue に反映しないと実装に落ちない。このスキルは **「フィードバック → Issue 更新」のギャップを埋める**。

| よくある状況 | このスキルがやること |
|------------|-------------------|
| 打ち合わせで方針が変わった | 関連 Issue の受け入れ条件・技術方針を更新 |
| 新しい要件が追加された | 既存 Issue に追記、または新規 Issue を作成 |
| 優先度が変わった | Issue のラベル・マイルストーンを変更 |
| 要件が削除・縮小された | Issue から該当部分を削除し、理由をコメント |
| 提案書が更新された | 提案書の変更点を対応する Issue に反映 |

## 実行フロー

```
引数チェック（空なら中断）
    │
Step 0: フィードバック解析
    │   ├─ フィードバック内容をパース
    │   ├─ 変更の種類を判定（追加/変更/削除/優先度変更）
    │   └─ ファイルパスがあれば Read で内容を取得
    │
Step 1: 現状把握 ──── ★2並列★
    │
    ├─ [並列A] 既存 Issue 一覧取得
    │   └─ gh issue list --state open --limit 100
    │
    └─ [並列B] 要件定義書の確認（あれば）
        └─ docs/requirements/ の関連ファイル
    │
    ▼ 結果統合
    │
Step 2: 影響範囲の特定
    │   ├─ フィードバックに関連する Issue を特定
    │   ├─ 各 Issue の現在の内容を取得
    │   └─ 影響マトリクスを作成
    │
Step 3: 変更案の作成 → ユーザー確認
    │   ├─ 既存 Issue の更新内容（差分）を提示
    │   ├─ 新規 Issue が必要な場合は案を提示
    │   ├─ 削除・クローズが必要な場合は理由を提示
    │   └─ 要件定義書の更新が必要な場合は案を提示
    │
Step 4: 反映実行
    │   ├─ Issue 本文の更新（gh issue edit）
    │   ├─ Issue コメントに変更履歴を追記
    │   ├─ 新規 Issue の作成（必要時）
    │   ├─ ラベル・マイルストーンの変更（必要時）
    │   └─ 要件定義書の更新（必要時）
    │
Step 5: 結果サマリー
        └─ 変更した Issue 一覧と変更内容を表示
```

## Step 0: フィードバック解析

### 入力のパース

引数テキストから以下を抽出する:

| 抽出項目 | 判定基準 | 例 |
|---------|---------|-----|
| **変更の種類** | キーワードで判定 | 「追加」「変更」「やめる」「優先度上げて」 |
| **対象の特定** | Issue番号、機能名、ファイルパス | 「#3 の件」「ログイン機能の件」「Phase 1」 |
| **変更内容** | フィードバックの本質 | 「ソーシャルログイン不使用」「通知連動追加」 |
| **理由・背景** | なぜ変更するか | 「レビュー指摘」「打ち合わせで決定」 |

### 変更の種類の判定

| 種類 | キーワード例 | Issue への操作 |
|------|------------|---------------|
| **要件追加** | 「追加」「〜も必要」「〜も含めたい」 | 受け入れ条件の追記 or 新規 Issue |
| **要件変更** | 「変更」「やっぱり」「〜ではなく」 | 受け入れ条件の書き換え |
| **要件削除** | 「やめる」「不要」「スコープ外に」 | 該当部分の削除 + コメント |
| **優先度変更** | 「先にやりたい」「後回し」「急ぎ」 | ラベル・マイルストーン変更 |
| **方針変更** | 「方針」「アプローチ」「技術」 | 技術メモ・設計方針の更新 |
| **情報追加** | 「補足」「追加情報」「参考」 | Issue コメントに追記 |

### ファイル入力の処理

引数にファイルパスが含まれる場合:

1. ファイルを Read で取得
2. 対応する既存 Issue を検索
3. ファイルの内容と Issue の差分を検出
4. 差分をフィードバック内容として処理

## Step 1: 現状把握（2並列）

### 並列A: 既存 Issue 一覧取得

```bash
# Open Issue の一覧
gh issue list --state open --limit 100 --json number,title,labels,body,milestone

# フィードバックのキーワードで検索
gh issue list --search "<キーワード>" --json number,title,body
```

### 並列B: 要件定義書の確認

```bash
ls docs/requirements/ 2>/dev/null
```

要件定義書がある場合は内容を読み取り、Issue との対応関係を把握する。

## Step 2: 影響範囲の特定

### 関連 Issue の特定方法

以下の優先順で関連 Issue を特定する:

1. **Issue 番号が明示されている**: `#3`, `Issue 3` → 直接参照
2. **機能名・キーワードで検索**: フィードバック内容のキーワードで Issue タイトル・本文を検索
3. **ファイルパスから逆引き**: 提案書・要件書に対応する Issue を検索
4. **Phase・マイルストーンから絞り込み**: 「Phase 1」等の言及があれば該当 Phase の Issue に絞る

### 影響マトリクスの作成

```
| Issue | タイトル | 影響の種類 | 変更概要 |
|-------|---------|-----------|---------|
| #3    | ユーザー認証 | 受け入れ条件の変更 | ソーシャルログイン不使用を明記 |
| #5    | 画面UI設計 | 設計方針の変更 | メール認証のみに変更 |
| (新規) | —       | 新規作成 | 通知連動機能 |
```

### 関連 Issue が見つからない場合

- フィードバック内容を要約し、新規 Issue として作成するか確認
- 要件定義書への追記のみで十分か確認

## Step 3: 変更案の作成 → ユーザー確認

### 提示フォーマット

```markdown
## フィードバック反映案

### 📝 フィードバック内容
{フィードバックの要約}

### 🔄 既存 Issue の更新

**#3 ユーザー認証プロトタイプ**
- 受け入れ条件に追加:
  + [ ] ソーシャルログインは使用しない（メール認証のみ）
- 技術メモに追記:
  + 運用コスト削減のため外部 IdP 連携は見送り（レビュー指摘）

**#5 画面UI設計**
- 設計方針を変更:
  - 変更前: ソーシャルログインボタンを含むレイアウト
  + 変更後: メール/パスワード入力フォームのみ

### ➕ 新規 Issue

**通知連動機能**
- 概要: イベント発生時にプッシュ通知を自動送信
- 受け入れ条件: ...
- 依存: #3（ユーザー認証）

### 📄 要件定義書の更新（該当する場合）
- docs/requirements/sample-feature.md
  - セクション 4.3 に VTR 連動を追記

---
この変更案でよろしいですか？ 修正があればお知らせください。
```

### ユーザー確認のポイント

- **必ずユーザー確認を取ってから実行する**（不可逆な操作を含むため）
- 修正が入った場合は変更案を更新して再提示
- 「全部OK」の場合のみ Step 4 に進む

## Step 4: 反映実行

### Issue 本文の更新

```bash
# Issue 本文を取得
gh issue view <number> --json body -q '.body'

# 更新した本文で上書き
gh issue edit <number> --body "$(cat <<'EOF'
<更新後の本文>
EOF
)"
```

### 変更履歴コメントの追記

**全ての更新 Issue に変更履歴コメントを追記する。** これにより「なぜ変わったか」が追跡可能になる。

```bash
gh issue comment <number> --body "$(cat <<'EOF'
## フィードバック反映

**日時**: 2026-04-04
**変更内容**: <具体的な変更>
**理由**: <なぜ変更したか>
**変更者のフィードバック**: <元のフィードバック内容>
EOF
)"
```

### 新規 Issue の作成

```bash
gh issue create --title "..." --label "..." --body "$(cat <<'EOF'
<Issue本文>

---
📝 この Issue は既存要件へのフィードバックから作成されました。
関連: #<元のIssue番号>
EOF
)"
```

### ラベル・マイルストーンの変更

```bash
# ラベル変更
gh issue edit <number> --add-label "priority:high"
gh issue edit <number> --remove-label "priority:low"

# マイルストーン変更
gh issue edit <number> --milestone "Phase 1"
```

### 要件定義書の更新

フィードバックが要件定義書にも影響する場合、`docs/requirements/` 内の該当ファイルを Edit ツールで更新する。

更新時のルール:
- 変更箇所にコメントで日付と理由を残す
- セクション構造は既存フォーマットを維持

## Step 5: 結果サマリー + 実行計画

反映完了後、**変更結果**と**次に実行すべき Issue の順序**を必ず提示する。

### 5-1: 変更結果

```markdown
## フィードバック反映完了

| 操作 | Issue | 内容 |
|------|-------|------|
| 更新 | #3 ユーザー認証 | 受け入れ条件にソーシャルログイン不使用を追加 |
| 更新 | #5 画面UI設計 | 設計方針をメール認証のみに変更 |
| 新規 | #12 通知連動 | 新規作成（#3 に依存） |
| 更新 | docs/requirements/sample-feature.md | セクション 4.3 更新 |

変更履歴コメントを各 Issue に追記しました。
```

### 5-2: 実行計画（★必須★）

フィードバック反映後の **全 Open Issue** を依存関係・優先度順に並べ、**次にどのチケットから着手すべきか**を明示する。

#### 実行計画の作成手順

1. **全 Open Issue を取得**: `gh issue list --state open --json number,title,labels,milestone`
2. **依存関係を解析**: Issue 本文中の `依存: #N` や `Blocked by #N` を解析
3. **優先度を判定**: ラベル（`priority:high` 等）、マイルストーン、依存関係から順序を決定
4. **実行順序を提示**: 依存関係のないものから順に、並列実行可能なものはグループ化
5. **推奨コマンド分岐**: 今回フィードバックで作成/変更された Issue のラベルを見て `/fix-impl` か `/issue-flow` かを判定

#### コマンド分岐ルール

| 条件 | 推奨コマンド | 理由 |
|------|-------------|------|
| 変更/作成された全 Issue が fix 系ラベル（`bug`/`refactor`/`documentation`/`test`/`performance`） | `/fix-impl` | 軽微な修正をバッチで並列実装、1つのPRにまとめる |
| いずれかに `enhancement`/`feature` を含む | `/issue-flow #N` | フル4フェーズ開発（分析→実装→レビュー→PR） |
| fix 系 と enhancement 混在 | `/fix-impl` → `/issue-flow` | 先に fix 系を片付けてから機能追加 |
| Issue が1件だけで小さな修正 | `/issue-flow #N` | 1件なら直接フローのほうが明確 |

**判定ロジック**:
```
今回のフィードバックで作成/変更された Issue のラベルを取得
  ├─ 全て fix 系のみ？
  │   ├─ YES + 2件以上 → /fix-impl を推奨
  │   └─ YES + 1件のみ → /issue-flow #N を推奨（単体なら直接）
  └─ enhancement/feature を含む？
      ├─ fix 系のみと混在 → /fix-impl → /issue-flow の順を推奨
      └─ enhancement/feature のみ → /issue-flow #N を推奨
```

#### 提示フォーマット

```markdown
## 実行計画

### 今すぐ着手可能（依存なし）
| 順序 | Issue | タイトル | 理由 |
|------|-------|---------|------|
| 1 | #1 | プロジェクト初期セットアップ | 全 Issue の前提 |
| 2 | #3 | ユーザー認証プロトタイプ | Phase 0 の中核。★今回のフィードバックで更新済み |

### 次に着手（上記の完了後）
| 順序 | Issue | タイトル | 依存先 |
|------|-------|---------|-------|
| 3 | #5 | 画面UI設計 | #1 完了後 |
| 4 | #12 | 通知連動 | #3 完了後。★今回新規作成 |

### 後続（Phase 2 以降）
| Issue | タイトル | 備考 |
|-------|---------|------|
| #8 | 本格QA知識ベース構築 | Phase 2 |
| #9 | 内覧会申し込みフォーム | Phase 2 |

---
次のコマンド:
  /issue-flow #1   ← まずここから（enhancement 系は /issue-flow、fix 系バッチは /fix-impl）
```

**fix 系バッチ修正の場合の例**:
```markdown
---
次のコマンド:
  /fix-impl   ← bug/refactor/docs 系の #3, #5, #8 を並列修正（1PR にまとめる）
```

#### 実行計画のルール

- **今回のフィードバックで変更・追加された Issue には ★ マークをつける**（何が変わったか一目でわかるように）
- **並列実行可能な Issue はグループ化して示す**（依存関係がなければ同時着手可能）
- **Phase / マイルストーンで区切る**（今やるべきものと後回しのものを明確に）
- **コマンド分岐ルール** に従って `/fix-impl` か `/issue-flow #N` を推奨
- fix 系のみ 2件以上 → `/fix-impl` を推奨（バッチ実装で効率化）
- enhancement/feature を含む → `/issue-flow #N` を推奨（フル開発フロー）
- フィードバックにより優先順位が変わった場合、**変更前後の順序の違いを明記する**

## 特殊ケースの処理

### フィードバックが曖昧な場合

フィードバック内容から変更内容を一意に特定できない場合、質問して明確化する:

```
フィードバック: 「ログイン機能の件、もうちょっとシンプルにしたい」

→ 質問:
「シンプルにしたい」とは、具体的にどの部分ですか？
1. 画面のUI（要素を減らす）
2. 機能（Phase 1 のスコープを絞る）
3. 技術構成（アーキテクチャを簡素化）
4. その他（具体的に教えてください）
```

### 複数の Issue にまたがるフィードバック

1つのフィードバックが複数の Issue に影響する場合、**影響マトリクスで全体像を示してから**一括で反映する。個別に反映すると整合性が崩れるリスクがある。

### Issue が存在しない場合

関連する Issue が1つも見つからない場合:
1. まず `/issue-create` で Issue を作成するか確認
2. Issue なしでも要件定義書への反映のみ行うか確認
3. メモとして `docs/` 配下に記録するか確認

### 提案書からの反映

引数にファイルパスが含まれ、それが提案書（`docs/proposals/`）の場合:

1. 提案書の内容を読み取る
2. 対応する既存 Issue を検索
3. **提案書に含まれていて Issue に含まれていない要件**を抽出
4. 新規 Issue として追加するか、既存 Issue に受け入れ条件として追記するかを提案

## 他のスキルとの連携

```
/requirement → 要件定義書を生成
      ↓
/issue-create → Issue 作成
      ↓
/feedback → フィードバックを Issue に反映（このスキル）
      ↓
/issue-flow #N → 開発
      ↓
/feedback → 開発中のフィードバックも反映可能
```

- `/requirement` → 初期の要件定義
- `/issue-create` → Issue の新規作成
- `/feedback` → 既存 Issue へのフィードバック反映（このスキル）
- `/issue-flow` → Issue ベースの自動開発
- `/plan` → フィードバック後の計画見直し
