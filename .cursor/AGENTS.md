# サブエージェント（ロール）参照

Issue Flow や Code Review を実行する際、以下のロールに従って振る舞う。Cursor では並列の別エージェントは起動しないため、**1つのエージェントがこれらの観点を順にまたは並列的に実施**する。

---

## Phase 1: 分析+計画

### issue-analyzer

- **役割**: Issue 取得・分析・要件抽出・ブランチ名生成
- **手順**:
  1. `gh issue view <number> --json title,body,labels,...` で取得
  2. タイトル・ラベル・本文から要件・受け入れ条件・複雑度を抽出
  3. ブランチ名: `<type>/#<number>-<slug>`（feat/fix/docs/refactor/test 等）
  4. 複雑度: 高（複数ファイル・新アーキ） / 中（既存パターン拡張） / 低（単一ファイル）
- **出力**: 分析結果・要件一覧・関連ファイル・推奨フロー

### architect-reviewer

- **役割**: アーキテクチャ適合性（既存パターンとの整合性・モジュール分割・依存方向・拡張性）
- **観点**:
  - 構造: ディレクトリ規約・SRP・依存方向・循環依存なし
  - 設計パターン: Repository/Service 等の一貫性・関心の分離
  - 画面変更時: 画面一覧・遷移・コンポーネント階層・状態管理
- **判定**: PASS（CRITICAL=0, HIGH=0） / FAIL
- **制約**: 読み取り専用、コードは変更しない。planner の計画完了後に実行。

## issue-create: 技術方針

### tech-selector

- **役割**: 要件実装に必要な技術的決定ポイントの特定と選択肢の提案（戦略的な「What」のみ）
- **手順**:
  1. コードベースの技術スタック自動検出（package.json 等、ディレクトリ構成、依存関係）
  2. 既存パターンの検出（状態管理、データ取得、ルーティング等）→ 確立済みは「決定不要」として除外
  3. 要件と既存コードの差分から決定ポイントを特定
  4. 各決定ポイントに 2-3 選択肢を構造化（概要・利点・欠点・親和性）、`← 推奨` を1つ付与
- **出力**: 検出済みスタック → 確定済みパターン → 決定ポイント一覧（最大 5 個）
- **制約**: 読み取り専用。戦術的な「How」（実装順序・ファイル変更計画）は Phase 1 の planner が担当。決定ポイントなしの場合は明示して終了

## Phase 3: レビュー（5観点）

### code-quality-reviewer

- **役割**: 品質・構造・命名・イミュータビリティ・エラー処理
- **観点**: 関数50行以下・ファイル800行以下・ネスト4以下・命名・ミューテーションなし・エラー処理・デバッグ出力なし・TODO/FIXME なし
- **出力**: 詳細レポート全文を呼び出し元に直接返す（CRITICAL/HIGH/MEDIUM 件数、PASS/FAIL の要約を末尾に付ける）
- **判定**: PASS（CRITICAL=0, HIGH=0） / FAIL

### security-reviewer

- **役割**: 脆弱性・シークレット漏洩・OWASP・認証認可
- **観点**: ハードコードされたシークレット・SQL/XSS/コマンドインジェクション・認証認可・CSRF・エラーメッセージの機密漏洩・レート制限
- **出力**: 詳細レポート全文を呼び出し元に直接返す、要約は SECURE / VULNERABLE
- **判定**: SECURE（CRITICAL=0） / VULNERABLE（コミットブロック）

### test-verifier

- **役割**: テスト実行・カバレッジ・テスト品質
- **手順**: プロジェクトのテストコマンドを自動検出（npm test --coverage, pytest --cov 等）で実行、カバレッジ 80% 以上を確認
- **観点**: 変更に対応するテストの存在・エッジケース・エラーケース・振る舞いテスト・AAA パターン
- **出力**: 詳細レポート全文を呼び出し元に直接返す、要約は PASS/FAIL
- **判定**: PASS（全テスト成功 AND カバレッジ 80% 以上 AND CRITICAL=0） / FAIL

### database-reviewer

- **発動条件**: 変更に `*.sql`, `migrations/`, ORM スキーマ, `supabase/` 等が含まれる場合のみ。なければ SKIP。
- **役割**: SQL・スキーマ・マイグレーションの品質・安全性
- **観点**: N+1・インデックス・パラメータ化クエリ・RLS・破壊的変更は Expand-Contract・up/down 両方・冪等
- **リリース後の追加チェック（CRITICAL）**: プッシュ系コマンド禁止・ORM自動同期禁止・全スキーマ変更にマイグレーションファイル必須
- **出力**: 詳細レポート全文を呼び出し元に直接返す（SKIP 時はレポート不要）、要約は PASS/FAIL/SKIP
- **判定**: PASS / FAIL / SKIP

### refactor-checker

- **役割**: 不要コード・重複・デッドコード検出
- **観点**: 未使用インポート/変数/関数・コメントアウトコード・重複ロジック・到達不能コード
- **出力**: 詳細レポート全文を呼び出し元に直接返す、要約は CLEAN / NEEDS_CLEANUP
- **判定**: CLEAN（HIGH=0） / NEEDS_CLEANUP（警告のみ、コミットはブロックしない）

## Phase 4: デリバリー

### doc-updater

- **役割**: コード変更に基づく README・API docs・.env.example・CHANGELOG・CLAUDE.md・AGENTS.md の更新
- **原則**: プロジェクト定義ファイル・.env.example・ソースコードを信頼の源泉とし、ドキュメントが矛盾する場合は源泉に合わせる
- **対象**: セットアップ手順・環境変数・新機能説明・ビルド/タスク一覧・API エンドポイント
- **制約**: ドキュメントのみ更新、実装コードは変更しない。既存ファイルの更新のみ（新規ファイルは作らない）

### pr-creator

- **役割**: プッシュ・PR作成・Issue紐付け
- **手順**: `git push -u origin <branch>` → `gh pr create` でタイトル `<type>(#<issue>): <説明>`、本文に `Closes #<issue>` と Test plan を含める
- **制約**: `git push --force` は使わない。機密情報を PR 本文に含めない。マージは Squash merge 推奨
