# Workflow オーケストレーション運用ルール

Claude Code の **Workflow（Dynamic Workflows）** 機能を acode ハーネスの中でどう使うかを定める。
基本思想は **「頭脳と品質管理＝ハーネス（CLAUDE.md・スキル・plan・review・progress）／大規模並列の手足＝Workflow」**。
Workflow に任せるのは **大規模並列実行とそのオーケストレーションだけ**。判断・計画・レビュー・永続化・最終品質チェックはハーネスが持つ。

## Workflow とは（前提）

- プロンプトに「workflow」と書く、または **スキルの指示が Workflow 呼び出しを authorize** すると、Claude が決定論的な JS オーケストレーションスクリプトを生成し、それに厳密に従って最大16同時・累計約1000のサブエージェントを並列実行する。
- 計画はコンテキストウィンドウではなく **スクリプト変数** に存在するため、超大規模タスクでもセッションが軽い。
- **再開可能**（中断地点から再開・完了エージェント結果はキャッシュ）・**監査証跡**（生成スクリプト＋エージェント別ログ）を備える。
- 生成されたスクリプト自身はファイル/シェルに触れない。読み書き・実行を行うのは生成された **エージェントのみ**（安全境界）。

## ★ オプトイン制約（最重要）

Claude は Workflow ツールを **ユーザーの明示的オプトイン時のみ** 呼べる。すなわち次のいずれか:

1. ユーザーがプロンプトに「workflow」と書く
2. **スキル/スラッシュコマンドの指示が Workflow 呼び出しを authorize する**（acode では `/orchestrate` スキルがこれを担う唯一の正規経路。組み込み `/workflows` は実行監視用で別物）
3. **ultracode が ON**（system-reminder で確認できる）。ultracode はそれ自体が standing opt-in の正規トリガで、substantive なタスクで Workflow を既定起動する。この場合 1 の「workflow」キーワードは不要。

→ **非 ultracode（high/max 等）では、ハーネスが勝手に Workflow を起動することはできない**。判断・提案は自動で行うが、起動の最終ゴーサインは必ずユーザーが出す。**ultracode 下は例外**: standing opt-in 自体がゴーサインに相当し、ハーネスは確認の往復を省いて起動してよい（下記「2つの確認ポイント」参照）。

## ★ 2つの確認ポイント（確認の往復であって、起動を禁じるハードブロックではない）

| # | タイミング | ハーネスのアクション |
|---|-----------|--------------------|
| ① 起動前 | 大規模と判断した直後 | `AskUserQuestion` で「Workflow を使うか / 従来の並列で進めるか」を**推奨第1候補つき**で提示し、承認を得る |
| ② 復帰前 | Workflow 完了後 | `AskUserQuestion` で「実行前の状態へ戻すか（ultracode 無効化・worktree 掃除等）」を提示し、承認を得てから復帰 |

①②は**確認の往復（1 回）**であって、Workflow の発火を止めるハードブロックではない。非 ultracode（high/max 等）では、判断は自動でも**起動と復帰は必ずユーザー確認を経る**（過剰消費・暴走・誤掃除の防止）。確認の作法は `decision-presentation.md` に従う。

### ultracode 下の扱い（★検証済みの事実）

ultracode 下では **① は standing opt-in ＋ ユーザーのタスク選択が代替する**ため、ハーネスは確認の往復を省いて直接起動してよい（実測: ある監査タスクで 28–31 エージェント並列が ① の `AskUserQuestion` なしで起動・完走）。**① を省いても fan-out もエージェント数も一切減らない＝ultracode の核（大規模並列）はそのまま発揮される**。仮に厳密適用しても増えるのは確認 1 往復だけで、能力低下はない。

→ ultracode 下は ① が外れるぶん、**実質の安全弁が「`while (budget.total && …)` の予算ガード必須」と「復帰②」の 2 つだけ**になる。とくに ultracode は token 目標未設定だと `budget.remaining()` が `Infinity` になり、予算ガードを書き忘れた動的ループが **1000 エージェント上限まで暴走**しうる（下記「並列の安全策」）。ここが ultracode 下で唯一残る実リスクなので、動的 workflow では必ず予算ガードを入れる。承認の往復を ultracode でも必ず効かせたい場合は、`/effort high` で起動する（standing opt-in が無いので ① が発火する）か、settings.json の `disableWorkflows` で明示制御する。

## 判定基準（ステップ1：Workflow 向きかの自動判断）★閾値の単一ソース★

**各スキルの委譲フックはここに集約された閾値を参照する**（スキル側に閾値の具体値を重複させない）。次のいずれかに該当したら「Workflow 向き」と判断し、確認ポイント①でユーザーに提案する:

- 変更/対象ファイルが概ね **50 以上**（実測: ユニークファイル件数）
- **複数モジュール / サービス横断**の変更（実測: トップレベルモジュール/ディレクトリ数 **≥ 3**）
- 大規模マイグレーション（言語・FW・スキーマの一括移行）
- サービス全体のリファクタリング
- **数十以上の独立サブタスク**（フラグ棚卸し・全 API 監査・全テスト生成 等。実測: ステップ数 **≥ 30**）
- 1 回の信頼できるパスでは大きすぎるバグハント / 包括テストスイート生成

### Workflow の才能が最も活きる入口（バッチ系）— 主軸

Workflow の本質は **「独立サブタスクが“多数”あるときの大規模ファンアウト」**。次の**バッチ入口**が主戦場であり、各スキルの委譲フックはここを最優先で狙う:

| 入口 | 委譲トリガー（バッチ専用閾値） |
|------|------------------------------|
| `/fix-impl`（fix Issue 一括） | 収集 Issue が概ね **10 件以上** |
| `/issue-flow --all-open`（複数 Issue 並列） | 対象 Issue が概ね **10 件以上** |
| `/code-review`・`/issue-flow` Phase 3（大規模 diff） | 上記ファイル/モジュール閾値、または敵対的検証が必要なとき |
| **リポジトリ全体監査**（`/code-review` → `exhaustive-review` `scope:"all"`） | 「全体を網羅監査」「セキュリティ全体チェック」等、diff ではなく追跡ファイル全体が対象のとき |
| **包括 E2E スイート生成**（`/e2e`） | クリティカルジャーニーが概ね 10 フロー以上 |

**非目標（自動フックを置かない）**: `/issue-flow` Phase 2 の**単一 Issue 実装**。単一 Issue が巨大化するのは分解不足であり、`/issue-create` での分解 → バッチ経路、または `/orchestrate <タスク説明>` の直接起動で扱う（自動委譲はしない）。

**いずれの閾値も未満なら Workflow を使わない**。従来の散文並列（`/issue-flow` Phase 1/3、`/code-review` の最大5並列、`/fix-impl` の最大5 Issue 並列）のままで十分であり、その方が軽量・低コスト。

## ハイブリッド7ステップの役割分担

| ステップ | 担当 | 使うもの |
|---|---|---|
| 1. タスク判断（Workflow 向きか） | **ハーネス** | 本ルールの判定基準 |
| 2. 事前分析・計画 | **ハーネス** | plan モード / `/plan` |
| 3. Workflow 起動指示 | **ハーネス** | `/orchestrate` スキル（authorize 役） |
| 4. オーケストレーションプラン生成→レビュー | Workflow生成 → **ハーネス**確認 | Workflow + ExitPlanMode |
| 5. 大規模並列実行 | **Workflow** | 定義済み `workflows/*.js` or 動的生成 |
| 6. 最終検証・品質保証 | **ハーネス** | `/code-review`・テスト実行 |
| 7. 永続化＋復帰 | **ハーネス** | progress 追記 + 復帰（後述） |

## モデルルーティング

- `agent()` の `model` は **原則省略**（セッションモデルを継承＝ほぼ正しい）。
- 明確にコスト/速度を改善できる場合のみ上書き: **探索・grep系・ファンアウト調査＝haiku**、**合成・敵対的検証・最終書き込み＝opus**。
- 最も高価なモデルで 100+ エージェントを回さない。

## 並列の安全策

- **worktree 分離**: 並列で**同じファイルを書き換える**場合のみ `isolation: 'worktree'`。読み取り中心や競合しない場合は使わない（~200-500ms/エージェントのコスト）。**例外（バッチ fix）**: `/fix-impl`・`fix-batch` の並列実装では影響ファイル集合の予測が**経験則で不正確になりうる**（共有 util・barrel/index・lockfile・生成物を予測外に触る）。予測が外れると共有作業ツリーで書き込みが静かに上書き/混線するため、**「非競合」と判定した並列レーン（グループ）でも worktree 分離する**。グループは互いにファイル素なのでバッチブランチへのマージは基本クリーンで、予測外の衝突は静かな破損でなく**明示的なマージ衝突**として顕在化する。
- 一時 worktree は `.claude/worktrees/` 配下に集約され、`/issue-flow` Phase 0 の自動掃除と同じ規約で片付ける（`git worktree unlock` → `--force` の2段防御）。
- **トークン予算ガード**: 動的ループは `budget.total` を必ずガードする。`budget.total` 未設定時は `budget.remaining()` が `Infinity` になり、`while (budget.remaining() > N)` が 1000 エージェント上限まで暴走する罠がある。`while (budget.total && budget.remaining() > N)` と書く。

## 観測性・監査

- 生成オーケストレーションスクリプト・エージェント別ログ・「何が反論されたか」・キャッシュ結果を**監査証跡として確認**する。
- 範囲を絞った場合（top-N・サンプリング・リトライ無し）は `log()` で「何を落としたか」を必ず明示（沈黙の打ち切りは「全部やった」と誤読される）。

## 段階導入

- いきなり全体に当てない。まず**サブディレクトリ / 単一モジュール**で検証してから全体へ。
- 長時間ジョブは**再開可能性**（resumeFromRunId）を前提に組む。

## ★ 復帰（クリーンアップ）— 確認付き自動復帰

Workflow（特に ultracode 下）は大量トークンを消費しうる。実行後は確認ポイント②を経て**実行前の状態へ戻す**。
復帰対象と、誰が実行できるかは次のとおり（**検証済みの事実に基づく**）:

| 対象 | 戻し方 | 実行者 |
|------|--------|--------|
| **ultracode** | `/effort high` で通常の high に戻す（`/effort ultracode` の解除）。settings.json キーは無く、**新セッションで自動リセット**される | `/effort` は**ビルトインのスラッシュコマンドで Claude からは直接実行不可**。→ ハーネスは**ユーザーに `/effort high` を打つよう案内**する（または「次セッションで自動的に戻る」と伝える） |
| **fast モード** | `/fast` トグル / settings.json `"fastMode": true` | `/fast` は Claude から直接実行不可。settings.json なら `update-config` 経由で編集可 |
| **一時 worktree** | `.claude/worktrees/` 配下の掃除 | **Claude が git で自動実行可**（決定論的） |
| **Workflow 機能の恒久無効化** | settings.json `"disableWorkflows": true` | 必要時のみ `update-config` で編集 |
| **Workflow 一時成果物/ログ** | 監査要約を progress に残してから掃除 | Claude が実行 |

> **実装メモ**: ultracode / fast の解除コマンドはアシスタントから直接打てないため、復帰は「**決定論的部分（worktree 掃除・progress 追記・必要なら settings.json 編集）は Claude が自動実行 ＋ ultracode 解除は正確なコマンド（`/effort high`）を提示してユーザーに打ってもらう**」という形になる。どのみち ultracode は新セッションで自動リセットされる点も併せて案内する。
