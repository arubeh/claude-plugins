# 画面構成 & 遷移設計

## 基本原則

**画面を含む機能は、実装前に画面一覧・遷移・コンポーネント構成を設計する。**

設計なしにコンポーネントを書き始めない。Phase 2（計画）で以下を明確にしてからコードに入る。

## 画面一覧の定義

画面を含む Issue では、実装計画に以下の形式で画面一覧を含める:

```
## 画面一覧

| # | 画面名 | パス | 説明 | 認証 |
|---|--------|------|------|------|
| 1 | ログイン | /login | メール+パスワード認証 | 不要 |
| 2 | ダッシュボード | /dashboard | 概要表示 | 必要 |
| 3 | ユーザー一覧 | /users | ページネーション付き | 必要 |
| 4 | ユーザー詳細 | /users/:id | 編集・削除操作 | 必要 |
| 5 | 404 | * | 存在しないパス | 不要 |
```

必須項目:
- **画面名**: ユーザーに見える名称
- **パス**: URL パターン（動的パラメータは `:param` 形式）
- **認証**: 認証要否

## 画面遷移の定義

画面間の遷移をテキストベースで明示する:

```
## 画面遷移

ログイン --[認証成功]--> ダッシュボード
ログイン --[認証失敗]--> ログイン (エラー表示)
ダッシュボード --[ユーザー管理クリック]--> ユーザー一覧
ユーザー一覧 --[行クリック]--> ユーザー詳細
ユーザー詳細 --[戻るボタン]--> ユーザー一覧
ユーザー詳細 --[削除確認]--> ユーザー一覧 (成功通知)
* --[未認証アクセス]--> ログイン
* --[存在しないパス]--> 404
```

記述ルール:
- `--[トリガー]-->` 形式でユーザー操作やイベントを明示
- 条件分岐がある場合はすべてのパスを列挙
- エラー時・未認証時の遷移も必ず含める
- ブラウザバック・リロード時の挙動も考慮

## コンポーネント階層の設計

画面ごとにコンポーネント構成を定義する:

```
## コンポーネント構成

ユーザー一覧 (/users)
├── Layout（共通レイアウト）
│   ├── Header（ナビゲーション）
│   └── Sidebar（メニュー）
├── UserListPage（ページコンポーネント）
│   ├── SearchBar（検索フィルター）
│   ├── UserTable（テーブル表示）
│   │   └── UserRow（行コンポーネント）
│   └── Pagination（ページネーション）
└── Footer（共通フッター）
```

### コンポーネント分類

| 種類 | 責務 | 配置例 |
|------|------|--------|
| **ページ** | ルーティング対応、データ取得 | `pages/` or `app/` |
| **レイアウト** | 共通構造（ヘッダー、サイドバー） | `layouts/` or `components/layout/` |
| **機能** | 特定ドメインのUI（UserTable等） | `features/<domain>/components/` |
| **共通UI** | 汎用部品（Button, Modal等） | `components/ui/` |

### 分割の判断基準

- 再利用される → 共通UIとして抽出
- 特定ドメインに属する → 機能コンポーネントとして `features/` 配下
- 1コンポーネント 200行超 → 分割を検討
- 状態管理が複雑 → コンテナ（ロジック）とプレゼンテーション（表示）に分離

## 状態管理の設計

画面に関わる状態を整理する:

```
## 状態設計

| 状態 | スコープ | 管理方法 | 例 |
|------|---------|---------|-----|
| サーバーデータ | グローバル | データフェッチライブラリ | ユーザー一覧 |
| フォーム入力 | ローカル | フォームライブラリ or useState | 編集フォーム |
| UI状態 | ローカル | useState | モーダル開閉 |
| 認証情報 | グローバル | Context / Store | ログインユーザー |
| URL状態 | URL | ルーター | ページ番号、フィルター |
```

原則:
- **URLに含められる状態はURLで管理する**（ページ番号、検索クエリ、フィルター）
- サーバーデータはデータフェッチライブラリ（SWR, TanStack Query, Apollo 等）で管理
- グローバルステートは最小限にする

## レスポンシブ対応

画面を含む機能では、対応範囲を明示する:

```
## レスポンシブ対応

| ブレークポイント | 幅 | 対応 |
|----------------|-----|------|
| モバイル | < 768px | 必須 / 不要 / 後日対応 |
| タブレット | 768px - 1024px | 必須 / 不要 / 後日対応 |
| デスクトップ | > 1024px | 必須 |
```

- 管理画面など内部ツール → デスクトップのみでも可
- ユーザー向け画面 → モバイル対応を原則必須とする
- 判断がつかない場合は Issue で対応範囲を確認する

## ローディング & エラー状態

すべての画面で以下の状態を設計する:

| 状態 | 表示内容 |
|------|---------|
| **ローディング** | スケルトン or スピナー |
| **空状態** | 「データがありません」+ アクション誘導 |
| **エラー** | エラーメッセージ + リトライボタン |
| **成功** | トースト通知 or インライン表示 |

```
# 各画面の状態設計例
ユーザー一覧:
  ローディング → UserTableSkeleton（スケルトン表示）
  空状態      → EmptyState（「ユーザーを追加してください」）
  エラー      → ErrorBanner（「取得に失敗しました」+ リトライ）
  成功        → UserTable（データ表示）
```

## FK参照フィールドのフォーム設計

**FKカラムのフォーム入力は、参照先の名称表示セレクトボックスを使うこと。UUID手入力は禁止。**

- DB スキーマに FK がある場合、フォームではセレクトボックス（ドロップダウン）で参照先を選択させる
- セレクトの表示ラベルは参照先テーブルの名称カラム（`name`, `last_name + first_name` 等）を使う
- UUID やIDをユーザーに直接入力させない（`<Input>` で UUID を入力するパターンは NG）
- セレクトの選択肢取得のため、参照先テーブルに一覧取得 API が存在することを前提とする
- 参照先の CRUD 画面・API が未実装の場合は、先にそちらを実装する

```
# NG: UUID手入力
<Input name="listingAgencyId" placeholder="元付不動産会社IDを入力" />

# OK: 名称表示のセレクトボックス
<Select name="listingAgencyId">
  {agencies.map(a => <SelectItem value={a.id}>{a.name}</SelectItem>)}
</Select>
```

## チェックリスト

画面を含む実装の設計時に確認:
- [ ] 画面一覧が定義されている（パス・認証要否を含む）
- [ ] 画面遷移が網羅されている（正常系・異常系・未認証）
- [ ] コンポーネント階層が設計されている
- [ ] 状態管理の方針が決まっている
- [ ] ローディング・空・エラー状態が考慮されている
- [ ] レスポンシブ対応範囲が明示されている
- [ ] 既存のコンポーネント・レイアウトパターンと一貫している
- [ ] FKカラムのフォーム入力がセレクトボックスになっている（UUID手入力でない）
