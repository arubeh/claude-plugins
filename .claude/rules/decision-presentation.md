# 意思決定の選択肢提示（難しい選択をわかりやすくする作法）

ユーザーに選択を求めるとき、**専門用語の壁で「どれを選べばいいか分からない」状態を作らない**ための作法。
`AskUserQuestion` を使う全場面（各スキルの「ユーザー確認」、方針判断、実装案の選択など）に適用する。

CLAUDE.md 回答スタイルの「推奨を第1候補」「自由入力より選択肢」「目的＋現在地を添える」を**具体的な選択肢の作り方に落とし込んだ拘束ルール**。矛盾する場合は本ルールを優先する。

## 適用対象

| 場面 | 適用 |
|------|------|
| 技術判断・方針判断・実装案の選択 | ✅ 必須 |
| 専門用語を含む / どちらが正解か非自明な選択 | ✅ 必須（特に丁寧に） |
| Yes/No・続行/中止の判断 | ✅ 必須 |
| ID 名・パス・固有値の入力 | ❌（選択肢化できないので自由入力で可） |

## 大原則（5つ）

1. **ラベルは平易な動詞で書く。** 「続行する」「ここで止める」「A 案で進める」。
   ラベルに専門用語（`mpsc::Sender`、`PgPool` 等）を入れない。専門語は description 側へ回す。

2. **各選択肢に「選ぶと何が起きるか」を1文で添える。** 得るもの・失うもの・次の一手のいずれかを、
   非専門の判断者でも評価できる言葉で書く。実装の手順説明（HOW）ではなく結果（WHAT）を書く。

3. **コスト・リスクを明示する。** 所要時間 / トークン消費 / 不可逆性 / 失敗時の影響 を、分かる範囲で。
   「volume 大」のような曖昧語ではなく「~600行・1セッションでは終わらない可能性」のように具体で。

4. **推奨を第1候補に置き、ラベル末尾に「(推奨)」＋理由1行。** 「迷ったらこれ」を必ず1つ作る。
   推奨できる根拠が無いなら、それ自体を正直に書く（「どちらも妥当・好みの問題」）。

5. **「こういう人/状況はこちら」の判断材料を添える。** 選択を相手の状況にマッピングする。
   例:「残り時間が少ない → B」「一気に仕上げたい → A」。

## 選択肢の構造テンプレート

各 option を次の3部構成で書く（label + description）:

```
label:       <平易な動詞。末尾に必要なら "(推奨)">
description: <①これを選ぶべき状況/人 → ②選ぶと何が起きるか(得る/失う/次) → ③コスト・リスク>
```

質問本文の直前（メッセージ側）には CLAUDE.md の規約どおり **目的＋現在地** を必ず添える。

## 提示前セルフチェック

`AskUserQuestion` を呼ぶ前に確認:

- [ ] ラベルから専門用語を排除し、平易な動詞にしたか
- [ ] 各選択肢に「選ぶと何が起きるか（WHAT）」を1文添えたか
- [ ] コスト/リスク/不可逆性を具体的な数値・条件で書いたか（曖昧語を排したか）
- [ ] 推奨を第1候補に置き、理由を1行添えたか（推奨不能ならその旨を明記したか）
- [ ] 「こういう状況はこちら」の判断材料を添えたか
- [ ] 質問の直前に目的＋現在地を添えたか

## Before / After 例

実装途中で「次のレベル(L5)に進むか、ここで止めるか」を問う場面。

### ❌ Before（専門用語の壁・判断材料なし）

> A. L5 (S9 ClasylogSubscriber 統合) に続行 ─ sqlx PgPool で id > last_seen polling、
> 3 source 並列ループ、mpsc::Sender で下流 tx、Backoff 適用… volume 大 (~600-800 行)。
> B. ここで停止 → 次セッションで /issue-flow #1 再起動して L5 から再開
> どちらにしますか?

問題点: ラベルが専門語の塊。所要コストが「volume 大」と曖昧。推奨が無い。
何を基準に選ぶのかが書かれていない。

### ✅ After

> **目的**: Layer A のログ取り込み機構を完成させること
> **現在地**: 部品実装(S0–S8 完了) → **統合層 L5(現在地)** → 統合テスト L6 → レビュー → PR
>
> | 選択肢 | 説明 |
> |--------|------|
> | **このまま L5 を実装する (推奨)** | 全部品が揃った今が組み立ての好機。3つのログ源を読み取り下流へ流す中核処理を作る。**コスト: ~600行・このセッションの残り context をほぼ使い切る見込み。テストは次の L6 で別途。** 一気に山場を越えたい人向け。 |
> | **ここで一旦止める** | 中核処理は次セッションに回す。`/issue-flow #1` で再起動すれば S8 完了地点から L5 を再開できる(進捗はチェックポイント済み)。**コスト: ほぼゼロ・失うものなし。** context を新鮮な状態で山場に臨みたい人向け。 |

ポイント: ラベルは動詞。中身は「何が起きるか」を平易に。コストを具体数値で。
推奨を先頭に理由付きで。「こういう人向け」で判断を後押し。

## 注意

- 選択肢を3〜4個に絞る（`AskUserQuestion` は最大4）。多すぎると逆に選べない。
- 推奨を盛るために事実を歪めない。リスクは正直に書く（CLAUDE.md「わからないことは正直に言う」）。
- 不可逆・外向きの操作（削除・公開・外部送信）を含む選択肢は、その旨をラベル/説明に明示する。
