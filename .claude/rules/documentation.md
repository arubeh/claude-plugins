# ドキュメント管理

## 対象ドキュメント

| ドキュメント | 読者 | 目的 | 更新責任 |
|-------------|------|------|---------|
| `README.md` | **人間**（新メンバー、外部開発者） | 手を動かすための情報（セットアップ・使い方） | doc-updater + 開発者 |
| `CLAUDE.md` | **Claude Code**（AI） | 正しい設計判断のための基盤 | doc-updater + 開発者 |
| `AGENTS.md` | **Cursor**（AI） | 正しい設計判断のための基盤 | doc-updater + 開発者 |

## ドキュメント間の境界

**同じ情報を複数のドキュメントに書かない。** 各ドキュメントは読者が異なるため、書くべき内容も異なる。

| 情報 | README.md（人間向け） | CLAUDE.md / AGENTS.md（AI向け） |
|------|----------------------|-------------------------------|
| セットアップ手順 | ✅ 詳細に | ❌ |
| Seed データ・ログイン情報 | ✅ | ❌ |
| トラブルシューティング | ✅ | ❌ |
| 設計方針＋WHY | ❌ | ✅ |
| アーキテクチャ判断の根拠 | ❌ | ✅ |
| 技術スタック | ✅ 表形式 | ✅ 設計意図付き |
| ディレクトリ構成 | ✅ 簡潔 | ✅ WHY 付き |
| コマンド一覧 | ✅ 全コマンド | ✅ 主要コマンドのみ |
| API エンドポイント | ❌（OpenAPI 参照と記載） | ✅ 数＋ルートパスのみ |
| 環境変数 | ❌（`.env.example` 参照） | ❌（`.env.example` 参照） |

## README.md 管理ルール

### 設計思想

README.md は **人間が読む唯一のエントリポイント**。
目的は「クローンしてから動かすまで」を最短で案内すること。
AI が読む設計情報（WHY・アーキテクチャ判断）は CLAUDE.md / AGENTS.md に書く。

### README.md に書くもの

- **クイックスタート**: clone → install → 起動 → 動作確認（コピペで完結）
- **前提条件**: 必要なツールとバージョン
- **Seed データ**: ログインアカウント、サンプルデータの概要
- **プロジェクト構成**: ディレクトリツリー（簡潔に）
- **コマンド一覧**: 全スクリプト（プロジェクト定義ファイルと同期）
- **開発サーバー URL**: ポート・URL 一覧
- **トラブルシューティング**: よくある問題と対処法

### README.md に書かないもの

- API エンドポイント一覧（→ OpenAPI ドキュメントへの参照のみ記載）
- 設計方針・アーキテクチャ判断（→ CLAUDE.md / AGENTS.md の責務）
- DB テーブル構造・Enum 値（→ スキーマファイルの責務）
- 頻繁に変わる件数・バージョン（→ ソースの責務）
- `.env` の全変数転記（→ `.env.example` の責務）

### 判定基準

**「新メンバーがこの情報なしに手が止まるか？」**
→ Yes なら書く。No ならソース・OpenAPI・CLAUDE.md への参照で十分。

## CLAUDE.md / AGENTS.md 管理ルール

### 設計思想

CLAUDE.md は毎ターンのシステムプロンプトに載る。全トークンが全ターンに影響する。
目的は **「正しい初手を打つための判断基盤」** を提供すること。
ツールで読める実装詳細（カラム定義、API 全表、画面 UI 詳細）は書かない。

### コンテンツ分類

| 分類 | CLAUDE.md に書く | 書かない（ソース参照） |
|------|-----------------|---------------------|
| 設計パターン＋WHY | パターン名と理由 | — |
| 存在マップ | 名前＋関係性（1-2行/グループ） | カラム定義、Enum 全値 |
| 規約（Conventions） | ファイル配置パターン、命名 | — |
| 権限モデル | 概要＋パターン | 個別エンドポイントの権限表 |
| コマンド | ビルド・テスト・起動 | — |
| API | エンドポイント数＋ルートパス | 全エンドポイント一覧表 |
| 画面 | ルート数＋App Router パス | 各画面の UI 詳細 |
| 環境変数 | `.env.example` への参照のみ | 全変数の転記 |
| ドメイン制約 | ビジネスルール要約 | — |

判定基準: **「Claude がこの情報なしに間違った設計判断をするか？」**
→ Yes なら書く。No ならソース参照で十分。

### 構成の順序

1. プロジェクト概要（何を・誰向けに）
2. 技術スタック
3. ディレクトリ構造
4. 設計方針＋WHY（判断の一貫性を担保）
5. Conventions（新規追加時のパターン）
6. 主要エンティティ（名前＋関係性＋設計意図。カラム詳細はスキーマ参照）
7. コマンド
8. 参照先（API ルート、画面ルート、スキーマ、OpenAPI 等のパス）

原則 → パターン → 具体例 の順。設計方針を先に読ませ、エンティティの理解に活かす。

### サイズ目標

- **200行以内**を目標（システムプロンプト毎ターンコスト考慮）
- 200行を超えたら、実装詳細が紛れていないか「コンテンツ分類」表で点検する
- 詳細は `docs/` 配下または `@ソースパス` で参照

### 更新タイミング

以下のいずれかに該当する変更を行った場合、更新を検討する:

- 新しいテーブルやエンティティを追加した
- アーキテクチャ上の重要な判断をした
- 新しいコマンドやツールを導入した
- 既存の記述が実態と乖離している

### 更新の原則

- **WHY 重視**: 設計意図や判断理由を残す
- **具体性**: 「適切に処理する」ではなく「try-catch で捕捉し Sentry に送信」
- **実行可能**: コマンドはコピペですぐ動く形で書く
- **参照優先**: 実装詳細はソースパスへの参照で代替する

### 書いてはいけないもの

- ソースから導出可能な一覧表（API 全エンドポイント、カラム全定義、Enum 全値）
- バージョン番号やテスト件数など頻繁に変わる値
- リンターやフォーマッターで済む規約
- `.env.example` の転記（参照パスのみ記載）
- 特定の個人にしか関係しない情報（`CLAUDE.local.md` へ）

## 共通ルール

- ドキュメントの信頼の源泉はソースコード・プロジェクト定義ファイル・`.env.example`
- 源泉と矛盾する場合、源泉を正とする
- 自動生成ドキュメント（TypeDoc, Swagger 等）は手動更新しない
