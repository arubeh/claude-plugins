# Requirement スキル

対話的に要件を深掘りし、`/issue-create` → `/issue-flow` パイプライン全体が **推測ゼロ** で動ける要件定義書を生成します。

## 使い方

```
/requirement 入居者向け家賃明細閲覧機能
/requirement 社内ナレッジ検索システムをゼロから作りたい
/requirement docs/rough-idea.md この走り書きを要件に仕上げたい
```

**引数は必須。** `$ARGUMENTS` が空の場合は使い方を案内して中断。

## 設計思想

この要件定義書は人間だけでなく、下流の全エージェントが消費する:

| 要件定義のセクション | 消費するエージェント | なぜ必要か |
|-------------------|-------------------|-----------|
| 目的・背景 | issue-create | Issue 本文の「背景」に転記 |
| アクター・ユースケース | planner | 画面遷移・API 設計の根拠 |
| 機能要件 | issue-create | Issue 分割の単位 |
| データモデル | planner, architect-reviewer | スキーマ設計・FK 整合性チェック |
| 非機能要件・制約 | tech-selector | 技術決定ポイントの絞り込み |
| スコープ外 | planner, issue-create | 余計な実装の防止・Phase 振り分け |
| 受け入れ条件 | TDD Phase 2 | RED テストの元ネタ |
| 用語集 | 全エージェント | 命名の一貫性 |

**書かないもの**: 技術スタック詳細（tech-selector が判定）、ファイル構成（planner が導出）、API 仕様詳細（planner が生成）。

## 実行フロー

```
引数チェック（空なら中断）
    │
Step 0: コンテキスト収集
    │   ├─ CLAUDE.md / AGENTS.md 読み取り
    │   ├─ 既存の docs/requirements/ 確認
    │   ├─ 引数にファイルパスがあれば読み込み
    │   └─ コードベース概要把握（既存PJの場合）
    │
Step 1: 入力パース＋充足度判定
    │   └─ 8セクション × 3段階（欠落/曖昧/十分）を判定
    │
Step 2-4: 適応的質問（最大3ラウンド）
    │   ├─ Round 1: WHO + WHY（アクター・目的）
    │   ├─ Round 2: WHAT（機能・データモデル・ルール）
    │   └─ Round 3: BOUNDARY（非機能・スコープ外・受け入れ条件）
    │
Step 5: ドラフト生成＋ユーザー確認
    │   └─ 全文を表示して修正・追加を受け付け
    │
Step 6: ファイル書き出し
    │   └─ docs/requirements/{slug}.md
    │
Step 7: 次のアクション案内
        └─ /issue-create docs/requirements/{slug}.md
```

## Step 0: コンテキスト収集

### プロジェクト状態の把握

```bash
# 既存の要件定義を確認
ls docs/requirements/ 2>/dev/null
```

- CLAUDE.md の `## Project Context` があれば技術スタック・既存パターンを把握
- 引数にファイルパス（`.md`, `.txt` 等）が含まれる場合は Read で内容を取得
- 既存 PJ の場合はコードベース構成を軽く調査（エンティティ名、ルート構成等）

### 既存要件との関係判定

| 状況 | 対応 |
|------|------|
| 既存要件なし | 新規作成 |
| 既存要件あり＋別領域 | 新規作成（既存を参照情報として利用） |
| 既存要件あり＋同領域 | ユーザーに確認: 上書き / 追補 / 中断 |

## Step 1: 入力パース＋充足度判定

ユーザーの入力（引数テキスト or 読み込んだファイル）を分析し、8セクションそれぞれの充足度を判定する。

### 8セクションと充足度基準

| # | セクション | 十分 | 曖昧 | 欠落 |
|---|-----------|------|------|------|
| 1 | 目的・背景 | Why が明確 | What はあるが Why がない | 言及なし |
| 2 | アクター | 名前＋役割が列挙 | 「ユーザー」のみ | 言及なし |
| 3 | ユースケース | 操作フロー記述あり | 機能名のみ | 言及なし |
| 4 | 機能要件 | 「〜できる」形式で3個以上 | 箇条書きだが曖昧 | 言及なし |
| 5 | データモデル | エンティティ＋関係 | エンティティ名のみ | 言及なし |
| 6 | 非機能要件 | 具体的な制約値あり | 「セキュリティに注意」等 | 言及なし |
| 7 | スコープ外 | 除外項目が明示 | — | 言及なし |
| 8 | 受け入れ条件 | テスト可能な条件あり | 「正しく動く」等 | 言及なし |

**判定結果に基づき、欠落セクションは必須質問、曖昧セクションは深掘り質問、十分セクションはスキップする。**

## Step 2: Round 1 — WHO + WHY（アクター・目的）

**前提**: Round 1 の回答がないと、Round 2 で機能要件を正しく聞けない。

### 質問項目（欠落/曖昧のもののみ）

**目的・背景が欠落の場合:**
> この機能はどんな課題を解決しますか？ 現状の問題点や、なぜ今これが必要かを教えてください。

**アクターが欠落の場合:**
> この機能を使う人は誰ですか？ 複数の役割がある場合はすべて挙げてください。
> （例: 入居者、管理会社スタッフ、システム管理者）

**アクターが曖昧の場合（「ユーザー」のみ）:**
> 「ユーザー」をもう少し具体化できますか？ 権限や操作範囲が異なる役割はありますか？

### 回答の反映

回答を受けて、セクション 1（目的・背景）と セクション 2（アクター）のドラフトを内部的に更新する。

## Step 3: Round 2 — WHAT（機能・データモデル・ルール）

**前提**: Round 1 で確定したアクターを使って「誰が何をできるか」を聞く。

### 質問項目（欠落/曖昧のもののみ）

**ユースケースが欠落の場合:**
> {アクターA} がこの機能を使うとき、どんな操作の流れになりますか？
> 主要なユースケースを 1-3 個、ステップで教えてください。
> （例: ログイン → 明細一覧 → 月選択 → 明細詳細 → PDF DL）

**機能要件が欠落/曖昧の場合:**
> 具体的にどんなことができる必要がありますか？ 「〜できる」の形で列挙してください。
> （Round 1 の回答から推測した候補を提示し、過不足を確認する）

**データモデルが欠落の場合:**
> この機能で扱う主なデータ（モノ・コト）は何ですか？ それぞれの関係性も教えてください。
> （例: 「物件」と「契約」は 1:N、「契約」と「入居者」は N:1）

**データモデルが曖昧の場合（名前だけある）:**
> {エンティティA} と {エンティティB} の関係はどうなりますか？（1:1 / 1:N / N:N）
> {エンティティA} で特に重要な属性や制約はありますか？

**ビジネスルールを検出した場合:**
> 以下の理解で合っていますか？ 他にビジネス上の制約やルールがあれば教えてください。
> （入力テキストから推測したルールを列挙して確認）

### 回答の反映

セクション 3（ユースケース）、4（機能要件）、5（データモデル）を更新。
ビジネスルールが出てきた場合はセクション 5b（ビジネスルール）として追加。

## Step 4: Round 3 — BOUNDARY（非機能・スコープ外・受け入れ条件）

**前提**: 機能が確定してから境界を引く。

### 質問項目（欠落/曖昧のもののみ）

**非機能要件が欠落の場合:**
> 以下の観点で特に気にすべきことはありますか？
> - パフォーマンス（同時ユーザー数、レスポンス時間等）
> - セキュリティ（認証方式、アクセス制御、データ保護）
> - 既存システムとの連携や制約

**スコープ外が欠落の場合:**
> 今回の範囲に **含めない** ものを明確にしたいです。
> 「やりたいけど後回し」「関連するけど別機能」があれば教えてください。
> （Round 2 で出てきた機能から、Phase 分割の候補を提示する）

**受け入れ条件が欠落/曖昧の場合:**
Round 2 で確定した機能要件から受け入れ条件のドラフトを自動生成し、ユーザーに確認する:
> 各機能の「完了条件」を以下のように考えました。過不足はありますか？
>
> **{機能A}:**
> - [ ] {アクター}が{操作}すると{結果}になる
> - [ ] {異常系: 未認証ユーザーは 401}
> - [ ] {境界値: 0件の場合は空配列}

**受け入れ条件の自動生成ルール:**
- 各機能要件に対して最低 2 条件（正常系 1 + 異常系/境界値 1）
- 「〜すると〜になる」のテスト可能な形式で書く
- 曖昧な表現（「適切に」「正しく」）は使わない

### 回答の反映

セクション 6（非機能要件）、7（スコープ外）、8（受け入れ条件）を更新。

## Step 5: ドラフト生成＋ユーザー確認

### ラウンド省略の判定

Step 1 で全セクション「十分」と判定された場合、質問ラウンドをすべてスキップして直接ドラフト生成に進む。部分的に十分な場合は該当ラウンドの質問数が減るだけで、ラウンド自体はスキップしない。

### 出力フォーマット

以下の構造でドラフトを生成し、ユーザーに全文を提示する:

```markdown
# {機能名}

> このドキュメントは `/issue-create` の入力として使用できます。
> `/issue-create docs/requirements/{slug}.md`

## 1. 目的・背景

{なぜ必要か。現状の課題と、この機能で解決すること。}

## 2. アクターとユースケース

### アクター

| アクター | 役割 | 主な操作 |
|---------|------|---------|
| {名前} | {役割の説明} | {主要操作の概要} |

### ユースケース

#### UC-1: {ユースケース名}
- アクター: {誰}
- トリガー: {何がきっかけで開始するか}
- 基本フロー:
  1. {ステップ}
  2. {ステップ}
  3. {ステップ}
- 代替フロー: {異常系・分岐}

## 3. 機能要件

### F-1: {機能名}
{説明文（1-2行）}

**受け入れ条件:**
- [ ] {テスト可能な条件}
- [ ] {テスト可能な条件}
- [ ] {異常系の条件}

### F-2: {機能名}
...

## 4. データモデル

### エンティティ関連図（テキスト）

```
{エンティティA} 1--N {エンティティB}
{エンティティB} N--1 {エンティティC}
```

### エンティティ定義

| エンティティ | 説明 | 主な関係 | 設計意図 |
|------------|------|---------|---------|
| {名前} | {何を表すか} | {→ 関連先} | {なぜこの構造か} |

### ビジネスルール
- {ルール1: 制約や計算ロジック}
- {ルール2}

## 5. 非機能要件

| カテゴリ | 要件 |
|---------|------|
| セキュリティ | {具体的な要件} |
| パフォーマンス | {具体的な要件} |
| 既存連携 | {具体的な要件} |

## 6. スコープ外

| 項目 | 理由 | 対応時期 |
|------|------|---------|
| {除外項目} | {なぜ今回含めないか} | {Phase N / 未定} |

## 7. 用語集

| 用語 | 定義 |
|------|------|
| {用語} | {このプロジェクトでの意味} |
```

### ユーザー確認

ドラフト全文を表示した後:

```
この要件定義でよろしいですか？
修正・追加があればお知らせください。問題なければファイルに書き出します。
```

修正が入った場合はドラフトを更新して再表示する。複数回の修正に対応する。

## Step 6: ファイル書き出し

### ファイルパス

```
docs/requirements/{slug}.md
```

`{slug}` は機能名から自動生成（ケバブケース、英語、30文字以内）。

```bash
# ディレクトリがなければ作成
mkdir -p docs/requirements
```

### 既存ファイルとの衝突

同名ファイルが存在する場合はユーザーに確認:
- 上書き
- 別名で保存（`{slug}-v2.md` 等）
- 中断

## Step 7: 次のアクション案内

```
要件定義を作成しました:
  docs/requirements/{slug}.md

次のステップ:
  /issue-create docs/requirements/{slug}.md この要件をもとに開発したい
```

## 質問戦略の詳細ルール

### 適応的省略

入力の情報量に応じて質問を省略する:

| 入力の情報量 | 質問ラウンド数 | 想定所要時間 |
|-------------|--------------|------------|
| 走り書き（1-3行） | 3ラウンド | 5-10分 |
| 中程度（段落レベル） | 1-2ラウンド | 3-5分 |
| 詳細（構造化済み） | 0ラウンド（確認のみ） | 1-2分 |

### 質問の形式

- **選択肢を提示できる場合**: 選択肢形式で負担軽減
  - 例: 「認証方式: A) NextAuth B) Supabase Auth C) 自前実装 D) その他」
- **自由回答が必要な場合**: 具体例を添えて質問
  - 例: 「どんなデータを扱いますか？（例: 物件→契約→入居者のような関係性）」
- **既存コードから推測できる場合**: 推測結果を提示して確認
  - 例: 「既存の User テーブルとの連携になりますか？」

### 1ラウンドあたりの質問数

**最大4問。** それ以上は認知負荷が高すぎる。優先度の高い欠落セクションから聞く。

## 出力品質チェックリスト

ドラフト生成前に以下を検証する:

- [ ] 全機能要件に受け入れ条件が付いている（最低2条件/機能）
- [ ] 受け入れ条件に曖昧な表現がない（「適切に」「正しく」を排除）
- [ ] データモデルの全エンティティに関係性と設計意図がある
- [ ] スコープ外セクションが空でない（明示的な境界がある）
- [ ] 用語集にドメイン固有の用語が含まれている
- [ ] 機能要件の粒度が Issue 分割可能（1機能 = 受け入れ条件 3-10個）

## 他のスキルとの連携

```
/requirement → 要件定義書を生成（このスキル）
      ↓
/issue-create docs/requirements/{slug}.md → Issue分割・作成
      ↓
/issue-flow #N → 自動開発
```

- `/requirement` → 要件定義書の生成（このスキル）
- `/issue-create` → 生成した要件定義書を入力として Issue 作成
- `/issue-flow #N` → Issue から自動開発
