# E2E スキル

ビルトインの **e2e-runner** エージェントを呼び出し、E2Eテストの生成・保守・実行を行います。プロジェクトに適したE2Eフレームワークを自動検出します（Playwright, Cypress, Selenium 等）。

## 大規模生成の委譲フック（上乗せ・既定挙動は不変）

クリティカルジャーニーが多数あり、**包括的な E2E スイートを一括生成**する場合（概ね 10 フロー以上＝`.claude/rules/workflow-orchestration.md` の「包括テストスイート生成」に該当）は、`/orchestrate` への委譲を**推奨第1候補で提案**する（`AskUserQuestion`）。委譲には Workflow のオプトインが必要なため、ユーザー承認のうえ起動する。定義済み workflow は無いため、orchestrate スキルが「フローごとに spec 生成→実行検証」を `pipeline()` で流す**動的 Workflow** を生成する（フロー一覧を scope として渡す）。**未満なら従来どおり e2e-runner 1 体のまま**（軽量）。

## 使い方

```
/e2e
/e2e ログインフロー
```

## 実行内容

1. **テストジャーニー生成** - ユーザーフロー用のE2Eテストを作成
2. **E2Eテスト実行** - プロジェクトのE2Eフレームワークでテストを実行
3. **アーティファクト取得** - 失敗時にログ、スクリーンショット等の証跡を取得
4. **レポート生成** - テスト結果レポートを出力
5. **不安定テスト特定** - 不安定なテストを隔離

## ベストプラクティス

**共通:**
- クリティカルなユーザージャーニーをテスト
- レスポンス/イベントの完了を待機（固定タイムアウトでなく）
- 実装の詳細ではなく振る舞いをテスト
- すべてのエッジケースを E2E でテストしない（単体テストを使用）

**Web UI テストの場合:**
- Page Object Model を使用
- セレクターに data-testid 属性を使用
- 脆弱なセレクターを避ける（CSSクラスは変わる）

**API テストの場合:**
- リクエスト/レスポンスのスキーマを検証
- 認証フローを含むシナリオをテスト
- エラーレスポンスのステータスコードと本文を検証

## 使用エージェント

| エージェント | 種類 | モデル |
|-------------|------|--------|
| e2e-runner | ビルトイン | inherit |

## クイックコマンド例

プロジェクトのE2Eフレームワークに応じてコマンドを使い分ける:

**Playwright (JS/TS/Python/Java/.NET):**
```bash
npx playwright test                    # 全テスト実行
npx playwright test --headed           # ブラウザ表示
npx playwright show-report             # レポート表示
```

**Cypress (JS/TS):**
```bash
npx cypress run                        # ヘッドレス実行
npx cypress open                       # インタラクティブ
```

**Selenium (Python):**
```bash
pytest tests/e2e/                      # pytest 経由で実行
```

**その他:** プロジェクトの設定ファイルからコマンドを自動検出。

## 他のスキルとの連携

- `/issue-flow` → Phase 3 のレビューと併用可能
- `/plan` → テストすべきクリティカルジャーニーを特定
- `/tdd` → 単体テスト（高速、粒度が細かい）
- `/code-review` → テスト品質を5並列レビューで確認
- `/build-fix` → テスト前にビルド成功を確認
